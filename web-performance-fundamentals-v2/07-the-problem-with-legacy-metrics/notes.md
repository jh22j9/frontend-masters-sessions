# Web Performance Fundamentals - 레거시 지표의 문제점

## 1. DOMContentLoaded·Load, 무엇이 문제인가?

DOMContentLoaded(DCL)와 Load는 한때 **웹 성능 측정의 유일한 수단**이었습니다.
시간의 흐름에 따라 두 수치를 기록하는 것이 사이트 속도를 파악하는 전부였고, 그 당시에는 이 방법이 잘 작동했습니다.

### 왜 잘 작동했나? — 서버 사이드 렌더링 시대

초기 웹에서 `load` 이벤트는 **문서가 준비(ready) 상태가 되었다**는 것을 의미했습니다.
jQuery 같은 초기 프레임워크의 코드도 이 가정을 기반으로 작성되었습니다.

```js
// 전통적인 jQuery 패턴
$(document).ready(function () {
  // DOM이 완성된 시점 → 비동기 작업 시작
  $.ajax("/api/cart", function (data) {
    $("#cart-count").text(data.count);
  });
});
```

서버가 완성된 HTML을 내려주는 구조에서는 `document ready` 시점이 곧 사용자가 콘텐츠를 볼 수 있는 시점이었기 때문에, DCL과 Load는 실제 사용자 경험을 잘 반영했습니다.

---

## 2. 2010년, 클라이언트 사이드 렌더링의 등장

**2010년**을 기점으로 **클라이언트 사이드 렌더링(Client-Side Rendering, CSR)** 이 등장했습니다.
Backbone, Knockout, jQuery UI 같은 프레임워크들이 이 방식을 초기에 구현했고, 이후 React·Vue 등 현대 프레임워크로 이어졌습니다.

### CSR 페이지의 전형적인 구조

```html
<!-- HTML 파일은 거의 비어 있음 -->
<html>
  <body>
    <div id="app"></div>
    <script src="app.js"></script>
  </body>
</html>
```

```js
// app.js — document는 거의 즉시 ready 상태가 됨
$(document).ready(function () {
  startApp(); // 이 시점부터 실제 렌더링 작업이 시작됨
});
```

HTML 자체에 콘텐츠가 없으므로 `document ready`는 **거의 즉시 발생**합니다.
실제 콘텐츠 렌더링은 JS 실행 이후에 이루어집니다.

---

## 3. CSR이 레거시 지표를 망가뜨린 이유

| 렌더링 방식 | DCL / Load 위치                              | 사용자 경험과의 연관성 |
| ----------- | -------------------------------------------- | ---------------------- |
| **SSR**     | 콘텐츠 로드 완료 부근                        | ✅ 의미 있음           |
| **CSR**     | 페이지 로딩 초반부 (HTML이 비어있으므로 즉시) | ❌ 의미 없음           |

GeoCities처럼 무거운 사이트를 CSR 방식으로 재구현하면:

- DCL과 Load는 **즉시(거의 0ms)** 발생
- 그러나 실제 사용자가 콘텐츠를 보기까지는 1초, 10초, 30초, 1분이 걸릴 수 있음
- 지표는 "빠르다"고 나타나지만 **사용자 경험은 최악**일 수 있음

> 즉, **지표가 빠를수록 좋은 사이트**라는 등식이 성립하지 않게 되었습니다.

## 4. Google이 새로운 지표를 필요로 한 이유

Google은 검색으로 수익을 창출합니다. 더 많은 사람이 검색을 사용하려면 **더 좋은 검색 결과**를 제공해야 하고, 좋은 검색 결과란 **빠른 웹사이트**와 연관됩니다.

Google의 고민:

> foo.com과 bar.com 중 어느 쪽이 더 빠른가?
> 콘텐츠 품질, 백링크가 동등하다면, 속도를 기준으로 순위를 매기고 싶다.

문제는 DCL과 Load를 기준으로 삼으면:
- SSR 사이트는 **실제보다 지표상 더 느리게** 보일 수 있음
- CSR 사이트는 **실제로는 느려도 지표상 빠르게** 보임

| | DCL | 실제 사용자가 콘텐츠를 보는 시점 |
|---|---|---|
| SSR | 800ms (느려 보임) | 800ms (일치) |
| CSR | 50ms (빨라 보임) | 3000ms (괴리 큼) |

따라서 Google은 다음 조건을 충족하는 새로운 측정 방법이 필요했습니다.

1. **사용자가 체감하는 성능**을 기반으로 측정
2. **프레임워크·기술 스택에 무관**하게 동작

---

## 5. Core Web Vitals의 등장

이러한 배경에서 등장한 것이 **코어 웹 바이탈(Core Web Vitals)** 입니다.

- 기존 DCL·Load의 한계를 극복하기 위해 새롭게 도입된 측정 기준 모음
- 사용자 인식 기반의 성능 측정 (렌더링 결과, 반응성, 시각적 안정성 등)
- SSR, CSR 등 구현 방식에 관계없이 실제 사용자 경험을 반영

> 다음 챕터에서 구체적인 Core Web Vitals 지표(LCP, INP, CLS)를 다룹니다.

## Q&A

<details>
<summary>jQuery는 CSR 프레임워크인가요?</summary>

jQuery는 CSR/SSR 프레임워크가 아닌 **DOM 조작 라이브러리**입니다.

jQuery는 `document`, `window` 등 브라우저 전역 객체에 의존하므로 Node.js 서버 환경에서 직접 실행할 수 없습니다. 그러나 **SSR + jQuery 조합은 가능**합니다. 서버가 완성된 HTML을 내려주고(SSR), 브라우저에서 jQuery가 인터랙션을 처리하는 방식입니다. 전통적인 PHP/Rails 앱이 대표적인 예입니다.

| | jQuery |
|---|---|
| 렌더링 방식 | 무관 (CSR/SSR 모두와 함께 쓸 수 있음) |
| 실행 환경 | 브라우저 전용 |
| 역할 | DOM 조작, 이벤트, AJAX 편의 API |

</details>

<details>
<summary>Google이 검색으로 수익을 창출한다는 것의 예시는?</summary>

검색 결과 페이지에 **광고(Search Ads)** 를 함께 보여주기 때문입니다. 사용자가 광고를 클릭하면 광고주가 Google에 비용을 지불하는 **CPC(Cost Per Click)** 구조입니다.

검색은 구매 의도가 명확한 순간이기 때문에 광고주 입장에서 전환율이 높고, 클릭당 높은 금액을 지불할 의향이 있습니다. Google 전체 매출의 약 75~80%가 광고에서 나오며, 그 대부분이 검색 광고입니다.

따라서 느린 검색 결과 링크로 사용자가 이탈하면 → 검색 자체를 덜 쓰게 되고 → 광고 수익이 줄어드는 구조이기 때문에, Google이 웹 성능에 관심을 가질 수밖에 없습니다.

</details>

<details>
<summary>백링크(Backlink)란 무엇인가요?</summary>

다른 웹사이트가 내 사이트를 링크로 가리키는 것입니다. 예를 들어 뉴욕타임스 기사에서 `foo.com`을 링크로 언급하면, `foo.com`이 백링크를 받은 것입니다.

Google의 초기 핵심 알고리즘(PageRank)은 **"많이 링크된 페이지 = 신뢰할 만한 페이지"** 라는 아이디어에서 출발했습니다. 백링크가 많을수록 검색 순위가 높아집니다.

노트에서 "콘텐츠 품질, 백링크가 동등하다면 속도를 기준으로 순위를 매기고 싶다"는 문장은, 기존 SEO 요소(콘텐츠·백링크)를 통제한 상태에서 **성능 지표를 추가 변수로 도입하겠다**는 의미입니다.

</details>

<details>
<summary>Google이 성능을 SEO 순위에 반영하기 시작한 건 언제부터인가요?</summary>

Google은 2021년 **Page Experience Update**를 통해 Core Web Vitals를 공식 SEO 랭킹 신호로 포함했습니다.
이전부터 사이트 속도를 랭킹 요소로 일부 반영해왔지만, Core Web Vitals가 그 기준을 더 구체화했습니다.

</details>

<details>
<summary>CSR은 어떻게 등장하게 됐나요?</summary>

특정 시점에 누군가 설계한 것이 아니라 **자연스럽게 진화한 패턴**입니다.

초기 웹은 서버가 모든 HTML을 만들어서 내려주는 구조였습니다. 페이지 전환마다 서버 요청 → 새 HTML 수신 → 전체 페이지 새로고침이 반복됐습니다.

**흐름:**
1. **AJAX 등장 (2005년경)** — 페이지 새로고침 없이 서버에서 데이터를 비동기로 받아오는 기술 보급
2. **JavaScript 역할 확대** — 데이터를 받아서 DOM을 직접 수정하는 코드가 늘어남
3. **SPA 아이디어** — "처음부터 빈 HTML 하나만 내려주고 JS가 모든 걸 그리면 어떨까?"로 발전
4. **프레임워크화** — Backbone.js(2010), AngularJS(2010) 등이 이 패턴을 구조화·대중화

Google Maps(2005)가 AJAX로 지도를 동적으로 로드하는 방식을 보여주며 "이런 UX가 가능하다"는 걸 증명한 것이 하나의 터닝포인트로 자주 언급됩니다.

</details>

<details>
<summary>Core Web Vitals는 SSR/CSR에 무관하다고 했는데, Next.js와 React를 비교하면 한쪽이 더 유리할 수도 있나요?</summary>

"구현 방식에 관계없이 측정한다"는 말은 **측정 방법이 공평하다**는 뜻이지, 결과가 같다는 의미는 아닙니다. 실제로는 **Next.js(SSR)가 Core Web Vitals에서 유리**합니다.

**LCP** — 가장 큰 차이가 나는 지표입니다.
- Next.js(SSR): 서버가 HTML에 콘텐츠를 담아 전송 → 브라우저가 바로 렌더링 → LCP 빠름
- React(CSR): 빈 HTML → JS 다운로드 → JS 실행 → API fetch → 렌더링 → LCP 느림

**CLS** — CSR은 데이터가 늦게 로드되며 레이아웃이 흔들릴 가능성이 높고, SSR은 초기 HTML에 콘텐츠가 있어 비교적 안정적입니다.

**INP** — 렌더링 방식보다 JS 실행 비용의 영향이 커서 SSR/CSR 차이보다 코드 최적화가 더 중요합니다.

| 지표 | Next.js(SSR) | React(CSR) |
|---|---|---|
| LCP | 유리 | 불리 |
| CLS | 유리 | 불리 |
| INP | 비슷 | 비슷 |

Core Web Vitals는 사용자가 실제로 체감하는 것을 측정하기 때문에, 구조적으로 더 빠른 SSR이 자연스럽게 좋은 점수를 받습니다.

</details>
