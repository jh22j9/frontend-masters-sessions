# JavaScript: The Hard Parts v3 - Arrow Functions & Built-in Methods

## 1. 현대적 함수 작성 스타일의 등장

클로저(Closure)로 넘어가기 전에, 함수를 작성하는 좀 더 **현대적인 스타일**을 짚고 넘어갈 필요가 있습니다. 동일한 `multiplyBy2` 함수를 **네 가지 서로 다른 버전**으로 단계별로 살펴보게 되는데, 모두 유효한 방식이며, 마지막에는 많이 쓰이는 **화살표 함수(Arrow Function) 스타일**에 도달합니다.

원래 문법은 단어가 많고 코드가 무거웠지만, 진화 과정을 거치면서 매우 읽기 쉽고 가독성이 좋은 형태로 바뀌었습니다.

---

## 2. 함수 선언의 4단계 진화

다음 네 줄은 모두 **동일한 `multiplyBy2` 함수**를 정의합니다. 각 줄은 서로 다른 파일에서의 독립적인 선언이며, 비교를 위해 나란히 놓은 것입니다.

```javascript
// 1) 전통적인 함수 선언 (function declaration)
function multiplyBy2(input) { return input * 2; }

// 2) 화살표 함수 + 중괄호 + return
const multiplyBy2 = (input) => { return input * 2; }

// 3) 중괄호와 return 키워드 제거 (암묵적 반환)
const multiplyBy2 = (input) => input * 2;

// 4) 파라미터가 하나면 괄호도 제거
const multiplyBy2 = input => input * 2;
```


| 버전  | 설명                          | 비고                          |
| --- | --------------------------- | --------------------------- |
| 1   | `function` 키워드를 사용한 전통적 선언  | 함수명이 `multiplyBy2`          |
| 2   | 화살표 함수로 상수에 할당              | `function` 키워드 생략           |
| 3   | 반환문이 하나뿐이면 중괄호와 `return` 생략 | **암묵적 반환(implicit return)** |
| 4   | 파라미터가 하나면 괄호도 생략            | 가장 간결한 형태                   |


> **여러 줄 함수 본문이 있다면?** 여전히 중괄호 `{ }`가 필요합니다. 중괄호와 `return` 생략은 반환문이 단 하나뿐일 때만 적용됩니다.

## 3. 내부 동작은 동일합니다

가장 축약된 4번 버전은 겉모습이 1번과 너무 달라서 "뭔가 특별한 거 아닐까?"라는 오해를 받기 쉽습니다. 그래서 **가장 낯선 형태를 대표로** 골라 내부 동작을 확인해봅니다.

네 번째 버전(`const multiplyBy2 = input => input * 2`)으로 `multiplyBy2(3)`을 호출하면, 이전 버전들과 완전히 똑같은 방식으로 동작합니다.

arrow-function-execution-context

```javascript
const multiplyBy2 = input => input * 2;
const output = multiplyBy2(3);
```

**실행 과정:**

1. `multiplyBy2`가 메모리에 저장됩니다 (함수 정의의 모습은 이전과 동일합니다).
2. `output` 변수가 선언되고, `multiplyBy2(3)`이 호출됩니다.
3. 새로운 **실행 컨텍스트(Execution Context)**가 생성됩니다. → **이후 4-5단계는 모두 이 실행 컨텍스트 내부에서 일어납니다.**
4. 파라미터 `input`이 값 `3`으로 설정됩니다. *(실행 컨텍스트의 로컬 메모리에 바인딩)*
5. `3 * 2` 즉 `6`이 반환됩니다. *(스레드 오브 실행이 코드를 실행)*
6. `output`에 `6`이 할당됩니다. *(반환 직후 실행 컨텍스트는 소멸)*

> **파라미터가 "실행 컨텍스트 안"에 저장된다는 사실이 왜 중요할까요?** 로컬 메모리는 **호출될 때마다 새로 만들어지는 공간**이므로 `multiplyBy2(3)`과 `multiplyBy2(5)`를 연달아 호출해도 서로 간섭하지 않고, 함수가 끝나면 `input`도 함께 사라집니다. 이 메커니즘이 나중에 **클로저(Closure)**를 이해하는 핵심 발판이 됩니다 — "함수가 끝나도 로컬 메모리가 살아남는다면?"이 바로 클로저의 출발점입니다.

> **핵심:** 화살표 함수는 단지 문법을 예쁘게 다듬은 것일 뿐이며, 실행 컨텍스트 생성·파라미터 할당·반환 등 내부 동작(under the hood)은 전통적 함수 선언과 동일합니다.

화살표 함수의 특별한 점 — `this` 바인딩

나중에 배우게 되지만, 화살표 함수는 한 가지 **독특한 동작**이 있습니다. 바로 **암묵적 파라미터 할당(Implicit Parameter Assignment)**, 즉 `**this` 키워드가 가리키는 대상**이 일반 함수와 다릅니다.

이 차이는 다른 함수 내부에서 함수를 호출할 때, 전역 객체(Global Object)가 아닌 **현재 다루고 있는 객체(Object)**를 참조하고 싶을 때 매우 유용합니다. 자세한 내용은 이후 챕터에서 다룹니다.

## 4. 익명 함수(Anonymous Function)와 인라인 전달

<b>익명 함수(Anonymous Function)</b>란 <b>이름이 없는 함수</b> 그 자체를 가리킵니다. 인자로 직접 넘기는 것은 익명 함수를 <b>사용하는 한 가지 방식</b>일 뿐이지, 정의는 아닙니다.

```javascript
// 1) 이름이 있는 함수로 전달
function multiplyBy2(input) { return input * 2; }
const result = copyArrayAndManipulate([1, 2, 3], multiplyBy2);

// 2) 화살표 함수를 상수로 저장한 뒤 전달
const multiplyBy2 = input => input * 2;
const result = copyArrayAndManipulate([1, 2, 3], multiplyBy2);

// 3) 익명 함수를 인라인으로 직접 전달 (이름 없음)
const result = copyArrayAndManipulate([1, 2, 3], input => input * 2);
```

세 번째 방식은 **Python의 람다 함수(Lambda Function)** 스타일에 가깝습니다. 전역(Global)에 이름을 부여하지 않고, 함수의 코드만 곧바로 전달합니다.

> **착각하지 말 것:** `input => input * 2`가 그 줄에서 실행되는 것처럼 보일 수 있지만, **실행되지 않습니다**. 단지 함수 정의가 `copyArrayAndManipulate` 내부의 `instructions` 자리에 전달될 뿐이고, 실행은 `instructions(array[i])`가 호출될 때 **새로운 실행 컨텍스트**를 만들며 이루어집니다.

Q: 상수에 저장하는 것과 인라인으로 전달하는 것 사이에 성능 차이가 있나요?

기술적으로는 미미한 차이가 있을 수 있고, 전체 코드베이스에 걸쳐 적용하면 이론상 이득이 될 수도 있습니다. 하지만 이는 **변수 이름을 짧게 하는 것**과 비슷한 수준의 극히 작은 차이입니다.

실제로 훨씬 더 큰 병목과 성능 이슈는 **개발팀으로서의 여러분의 시간**입니다. 그래서 언제나 더 중요한 것은 **코드의 가독성과 이해하기 쉬움**입니다.

> **강사의 핵심 메시지:** 개발 프로세스에서 점점 더 가장 가치 있는 자산은 **엔지니어로서의 여러분 자신**이지, 변수/상수 할당에서 오는 미미한 성능 개선 같은 것이 아닙니다.

## 5. 내장 메서드 `map`

`copyArrayAndManipulate`는 사실 너무 중요해서 JavaScript에 **내장(built-in)**되어 있습니다. 바로 `**map`**입니다.

`map`은 모든 프로그래밍 언어에서 가장 중요한 함수 중 하나로 꼽힙니다:

- 어떤 데이터를 가져다가, 각 요소에 대해 어떤 작업(수학 연산, UI 표시 등)을 수행합니다.
- **원본 데이터는 변경하지 않고**, 각 요소에 작업을 적용한 새로운 배열을 반환합니다.

> **예시 상황:** Spotify 노래 목록이 있을 때, 원본 노래 리스트는 그대로 두고 각 노래를 표시하는 **UI 요소 목록**을 만들고 싶은 경우.

### 직접 구현 vs 내장 버전

```javascript
// 1) 우리가 직접 만든 버전 (이름만 map으로 변경)
const result = map([1, 2, 3], input => input * 2);

// 2) JavaScript 내장 map 메서드 사용
const result = [1, 2, 3].map(input => input * 2);

// 두 경우 모두 결과는 [2, 4, 6]
```


| 구분          | 호출 형태            | 배열 전달 방식                            |
| ----------- | ---------------- | ----------------------------------- |
| 직접 만든 `map` | `map(array, fn)` | 명시적으로 첫 번째 인자로 전달                   |
| 내장 `map`    | `array.map(fn)`  | **점(dot) 표기법**을 통해 암묵적으로 왼쪽 배열이 전달됨 |


두 방식은 **완전히 동일한 일**을 수행합니다. 단지 내장 버전은 **프로토타입 체인(Prototype Chain)**을 통해 모든 배열에서 `.map` 형태로 사용할 수 있다는 점이 다릅니다.

내장 배열 메서드 목록은 어디서 찾나요?

**MDN 문서(MDN Docs)**가 표준적인 참조처입니다. 모든 배열은 `map`, `filter`, `reduce` 등 다양한 내장 메서드를 기본으로 제공합니다. `forEach`, `find`, `some`, `every` 등도 많이 쓰입니다.

배열이 메서드를 가질 수 있는 이유는? (선행 예고)

JavaScript에서는 **모든 것이 객체**이고, 객체는 프로퍼티를 가질 수 있습니다. 배열 역시 객체이며, **프로토타입 체인(Prototype Chain)**을 통해 내장 메서드들을 상속받습니다. 점(dot) 뒤에 붙는 `.map`, `.filter` 등이 바로 이러한 프로토타입 체인 메커니즘으로 배열에서 접근 가능해집니다. 자세한 내용은 OOP 챕터에서 다룹니다.

## 6. 요약: 가독성을 높이는 두 가지 도구


| 도구                                    | 역할           | 내부 동작 변화                             |
| ------------------------------------- | ------------ | ------------------------------------ |
| **화살표 함수(Arrow Function)**            | 함수 선언을 간결하게  | 단순히 예쁘게 만드는 것일 뿐 (단, `this` 바인딩은 다름) |
| **내장 배열 메서드(Built-in Array Methods)** | 배열 조작을 선언적으로 | 점(dot) 표기법으로 호출 가능                   |


**익명 함수 + 화살표 함수 + 내장 배열 메서드**의 조합은 특히 **함수형 프로그래밍(Functional Programming)**에서 데이터에 대한 일련의 작업을 체인(Chain)으로 연결할 때 강력합니다.

```javascript
// 작업 체인 예시
const result = [1, 2, 3]
  .map(x => x * 2)
  .filter(x => x > 2);
```

> **주의:** 이처럼 간결한 체인은 "실제로 내부에서 무슨 일이 일어나는지"를 가리기도 합니다. 그래서 내부 동작을 정확히 이해하는 것이 혼동을 피하는 데 필수적입니다.

### 강사의 개인적 관점

- **별도 정의 선호:** 함수의 동작을 명확히 보여주기 위해 별도로 이름 붙여 정의하는 것을 선호합니다.
- **체인의 현실:** 하지만 함수형 프로그래밍에서는 일련의 변환을 **인라인으로 전달**하는 경우가 많습니다.
- **추가 학습 추천:** 데이터에 대한 작업 체인에 대해 더 깊이 배우려면 *Functional Programming Hard Parts*를 시청할 것을 권장합니다.

## Q&A

### Q: "인라인으로 변환을 전달한다"는 게 FE(React) 관점에서 어떤 모습이고, 함수형이 아니면 어떻게 쓰나요?

<b>상황:</b> 상품 목록에서 재고 있는 것만 골라 리스트로 렌더링

<b>✅ 함수형 — 인라인 전달 (React 표준)</b>

```jsx
<ul>
  {products
    .filter(p => p.inStock)
    .map(p => <li key={p.id}>{p.name}</li>)}
</ul>
```

- `p => p.inStock`, `p => <li ...>` 같은 익명 함수를 <b>그 자리에서 바로 전달</b>
- 데이터가 `filter → map`을 <b>파이프처럼 통과</b>하며 새 배열로 변환됨
- 원본 `products`는 <b>절대 건드리지 않음</b> (불변성)

<b>🔴 함수형이 아닌 — 명령형(Imperative) 스타일</b>

```jsx
// jQuery 시절 또는 함수형을 모를 때의 전형적 코드
const items = [];                        // 1) 빈 배열 준비
for (let i = 0; i < products.length; i++) {   // 2) 루프
  if (products[i].inStock) {             // 3) 조건 체크
    items.push(<li key={products[i].id}>{products[i].name}</li>);  // 4) 직접 추가(mutate)
  }
}
return <ul>{items}</ul>;
```

- 빈 배열을 만들고 `push`로 <b>하나씩 직접 채워넣음</b> (mutation)
- "어떻게" 할지 <b>단계별로 지시</b>
- 의도(재고 있는 것만 렌더링)가 루프와 조건문에 <b>파묻힘</b>

<b>핵심 차이 한눈에:</b>

| 관점 | 함수형 (인라인 전달) | 비함수형 (명령형) |
|---|---|---|
| 사고방식 | "데이터를 <b>무엇으로</b> 변환할까" | "<b>어떻게</b> 단계별로 만들까" |
| 코드 | `filter().map()` 체인 | `for` + `if` + `push` |
| 원본 데이터 | 건드리지 않음 (새 배열 반환) | 새 배열에 직접 `push` |
| 함수 전달 방식 | 익명 함수를 <b>인라인</b>으로 | 루프 본문에 로직을 <b>인라인</b>으로 (함수 전달 아님) |
| React와의 궁합 | JSX와 자연스럽게 맞물림 | JSX 안에서 쓰기 어색함 |

<b>결론:</b> React 개발자가 `{items.map(...)}`, `setTodos(todos.filter(...))`처럼 매일 쓰는 그 코드가 바로 "일련의 변환을 인라인으로 전달하는" 함수형 패턴입니다. 반대로 `for` 루프 + `push`로 배열을 조립하는 옛 방식이 "함수형이 아닌" 스타일이고, 모던 React에서는 거의 사라졌습니다.

### Q: 그럼 함수형 = 선언형인가요?

<b>정확한 관계: 함수형 ⊂ 선언형</b> (함수형은 선언형의 <b>한 갈래</b>)

<b>두 축은 서로 다른 질문에 답합니다:</b>

| 축 | 질문 | 대립 쌍 |
|---|---|---|
| <b>표현 방식</b> | "<b>무엇을</b> 원하는가" vs "<b>어떻게</b> 만들 것인가" | 선언형 ↔ 명령형 |
| <b>코드 구조</b> | 코드를 무엇으로 조직하는가 | 함수형 ↔ OOP ↔ 절차적 |

<b>한눈에 보는 포함 관계:</b>

```
┌──────────── 선언형 (Declarative) ────────────┐
│                                              │
│   ┌─── 함수형 ───┐                            │
│   │ map/filter/ │   SQL, HTML/JSX, CSS,       │
│   │ reduce 체인 │   정규식, YAML/JSON 등        │
│   └─────────────┘   (선언형이지만 함수형 아님)    │
│                                              │
└──────────────────────────────────────────────┘

┌──────────── 명령형 (Imperative) ─────────────┐
│   for 루프 + push, jQuery DOM 조작,            │
│   class + this.state 변경                     │
└──────────────────────────────────────────────┘
```

<b>핵심 정리:</b>

| 질문 | 답 |
|---|---|
| 함수형 = 선언형? | ❌ 완전히 같진 않음 |
| 모든 함수형이 선언형인가? | ✅ 사실상 그렇다 |
| 모든 선언형이 함수형인가? | ❌ SQL·HTML·CSS·정규식은 선언형이지만 함수형 아님 |

<b>FE 실무 기억법:</b>

- <b>"무엇을 원하는가"</b>를 쓰면 → 선언형
- <b>"함수로 데이터를 변환"</b>하면 → 함수형
- React는 <b>JSX(선언형) + hooks/불변 상태(함수형)</b>의 조합

JavaScript 코드 안에서는 <b>함수형 코드 ≈ 선언형 코드</b>로 봐도 실무상 거의 맞습니다. 다만 이론적으로는 <b>선언형이 더 큰 우산</b>이라는 점만 기억하세요.

---

<details>
<summary>"함수형"이라는 이름은 정확히 무슨 뜻인가요?</summary>

"functional"의 어원은 일상 영어 "기능적인"이 아니라 <b>수학의 "함수(function)"</b>입니다. 한국어 "기능"이라는 뉘앙스는 오해를 유발하므로 잊어버리는 편이 좋습니다.

<b>각 패러다임 이름은 "무엇을 중심으로" 프로그램을 만드는지 가리킵니다:</b>

| 패러다임 | 중심 재료 |
|---|---|
| 절차적(Procedural) | 절차(procedure) — 단계별 명령 |
| 객체지향(OOP) | 객체(object) — 상태+행동 덩어리 |
| <b>함수형(Functional)</b> | <b>함수(function)</b> — 입력을 출력으로 변환하는 것 |

<b>여기서 말하는 "함수"는 수학적 함수입니다:</b>

- 같은 입력이면 <b>언제나 같은 출력</b> (`f(3)` = 항상 `6`)
- 바깥 세상을 건드리지 않음 (부수 효과 없음)

```javascript
// ✅ 수학적 함수에 가까움 — 함수형에서 이상적
const double = x => x * 2;

// ❌ 수학적 함수가 아님 — 바깥을 건드리고, 호출마다 결과가 다름
let count = 0;
function impure() { count++; return Math.random(); }
```

<b>한 문장 정리:</b>

> <b>함수형 = 프로그램의 빌딩블록으로 "수학적 함수"를 선택한 스타일</b>

수학적 함수는 예측 가능하고, 테스트하기 쉽고, 조합하기 편합니다. React가 불변성과 순수 함수를 강조하는 이유이기도 합니다.

</details>

## Quiz

**Q1.** 다음 네 가지 `multiplyBy2` 선언 중 **동작이 다른 것**은?

```javascript
// a)
function multiplyBy2(input) { return input * 2; }
// b)
const multiplyBy2 = (input) => { return input * 2; }
// c)
const multiplyBy2 = (input) => input * 2;
// d)
const multiplyBy2 = input => input * 2;
```

<details>
<summary>정답 보기</summary>

<b>모두 동일하게 동작합니다.</b> 네 버전 모두 `multiplyBy2(3)`을 호출하면 `6`을 반환하며, 실행 컨텍스트 생성 과정도 완전히 같습니다. 단, 화살표 함수는 `this` 바인딩이 다르다는 미묘한 차이가 있습니다.

</details>

<details>

<b>함수 본문이 단 하나의 반환문으로 구성되어 있을 때</b>만 생략할 수 있습니다. 여러 줄의 로직이 있다면 반드시 중괄호가 필요하며, 명시적으로 `return`을 써야 합니다.

</details>

**Q3.** 다음 중 **익명 함수(Anonymous Function)**의 예시는?

a) `function multiplyBy2(input) { return input * 2; }`  
b) `const multiplyBy2 = input => input * 2;`  
c) `copyArrayAndManipulate([1, 2, 3], input => input * 2);`  
d) `const result = multiplyBy2(3);`

<details>
<summary>정답 보기</summary>

<b>c) `copyArrayAndManipulate([1, 2, 3], input => input * 2);`</b>

익명 함수란 <b>이름이 없는 함수</b>를 말합니다. 인자로 직접 전달되는 경우가 대표적 사용 방식입니다. b의 경우 `multiplyBy2`라는 이름(레이블)에 할당되므로 엄밀히 익명 함수가 아니라고 볼 수 있습니다.

</details>

**Q4.** 다음 두 코드의 차이에 대해 올바른 설명은?

```javascript
// A
const result = map([1, 2, 3], input => input * 2);
// B
const result = [1, 2, 3].map(input => input * 2);
```

a) A와 B는 완전히 다른 동작을 한다  
b) A는 커스텀 `map`, B는 내장 `map` 메서드를 호출하며, **결과는 같다**  
c) B가 A보다 항상 빠르다  
d) A는 원본 배열을 변경하지만 B는 변경하지 않는다

<details>
<summary>정답 보기</summary>

<b>b) A는 커스텀 `map`, B는 내장 `map` 메서드를 호출하며, 결과는 같다</b>

두 방식 모두 `[2, 4, 6]`을 반환합니다. 내장 `map`은 프로토타입 체인을 통해 배열에서 점(dot) 표기법으로 호출 가능하며, 호출된 배열이 암묵적으로 첫 번째 인자로 전달됩니다.

</details>

**Q6.** `map` 함수의 핵심 특성에 해당하지 **않는** 것은?

a) 각 요소에 동일한 작업을 적용한다  
b) 원본 배열을 변경한다  
c) 새로운 배열을 반환한다  
d) 수학 연산, UI 요소 생성 등 다양한 목적에 쓰인다

<details>
<summary>정답 보기</summary>

<b>b) 원본 배열을 변경한다</b>

`map`은 <b>원본 데이터를 변경하지 않습니다</b>. 각 요소에 작업을 적용한 결과로 <b>새로운 배열</b>을 반환합니다. 이는 함수형 프로그래밍의 <b>불변성(Immutability)</b> 원칙과 맞닿아 있습니다.

</details>
